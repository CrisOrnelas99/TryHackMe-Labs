🔐 Authentication Bypass — Condensed Detailed Notes
📘 What Authentication Bypass Is

Authentication bypass refers to methods used to defeat or manipulate login and identity verification systems in order to gain access to restricted areas without proper authorization. These weaknesses are among the most serious in web security because they can lead directly to account compromise, unauthorized access to user data, privilege escalation, and exposure of sensitive information. Authentication flaws do not always come from weak passwords alone; they can also result from poor error handling, insecure logic, predictable workflows, and unsafe trust in client-controlled data. The main concepts covered here are username enumeration, brute force attacks, logic flaws, and cookie tampering, along with supporting ideas such as hashing and encoding.

🔎 Username Enumeration

Username enumeration is the process of identifying valid account names by observing how an application responds to authentication-related actions. This often happens when a website reveals different error messages depending on whether a username exists. For example, a registration form may respond with a specific message if a chosen username is already in use, which allows an attacker to confirm that the account exists. This is valuable because a list of valid usernames makes later attacks more focused and efficient. Rather than guessing both usernames and passwords blindly, an attacker can first confirm valid accounts and then target them with password attacks or reset abuse. In this case, automated requests were used to test many usernames against the signup form and identify which ones already existed based on the response text. The valid usernames discovered were simon, steve, and robert.

⚙️ Brute Force Attacks

A brute force attack is an automated process that repeatedly tries passwords against one or more accounts until a valid combination is found. In web authentication testing, brute force is often more effective when combined with username enumeration because it narrows the attack to confirmed valid users. Instead of wasting requests on non-existent usernames, the attack can focus only on accounts known to be real. Automation tools can pair a username list with a password list and test many combinations quickly against the login form. This demonstrates a common risk in weak authentication systems: if the application does not enforce rate limiting, account lockout, MFA, or anomaly detection, repeated login attempts may eventually succeed. The valid credential pair identified here was steve/thunder.

🧠 Logic Flaws in Authentication

A logic flaw is a weakness in how an application’s process is designed rather than a simple coding bug such as broken syntax or memory corruption. In authentication, logic flaws happen when the intended flow of identity checks can be bypassed, manipulated, or reordered to produce an unintended result. These issues are dangerous because the system may appear secure during normal use while still allowing an attacker to abuse the underlying logic. One example shown is a case-sensitive path check, where access control only protects requests that match an exact path such as /admin, but a variation like /adMin avoids the check and still reaches the page. The broader principle is that security logic must account for all equivalent forms of user input and cannot rely on narrow comparisons that attackers can work around.

A more practical authentication logic flaw appears in the password reset process. The reset workflow first accepts an account email and then asks for the related username. At a glance, this seems secure because both pieces of information must match before a reset link is sent. However, the weakness comes from how the application processes user-supplied data. The account lookup uses the email in the query string, but the reset delivery later relies on the PHP $_REQUEST variable, which combines query string and POST data and gives preference to POST values if the same parameter name appears in both places. Because of this, an attacker can provide the legitimate victim email in the URL for account validation while supplying a different email in the POST body for delivery. This changes where the reset message is sent without changing which account is being targeted. As a result, the reset link for Robert’s account could be redirected to the attacker-controlled support email, allowing access to Robert’s account and his support tickets. The flag obtained through this authentication bypass was THM{AUTH_BYPASS_COMPLETE}. This is a strong example of why security decisions must use a single trusted source of input and must not mix parameters from multiple locations without strict validation.

🍪 Cookie Tampering

Cookie tampering is the act of modifying values stored in browser cookies in order to alter application behavior or gain unauthorized access. Since cookies are stored on the client side, they should never be trusted to enforce sensitive security decisions unless they are strongly protected and validated server-side. If an application stores privilege information directly in a readable or modifiable cookie, an attacker may simply change the value and resend it with future requests. The example given shows plain-text cookies such as logged_in=true and admin=false. By changing these values manually, the user can alter how the server interprets their session state. When both cookies were set to true, the application treated the request as an administrator session. The flag gained from changing the plain-text cookie values was THM{COOKIE_TAMPERING}. The main lesson is that authorization decisions must not depend on client-editable state without secure integrity protection.

#️⃣ Hashing and Encoding

Hashing and encoding are both used to transform data, but they serve very different purposes. A hash is a one-way transformation that produces a fixed representation of the original input. It is designed to be irreversible in normal use, meaning the original value should not be directly recoverable from the hash alone. Hashes are commonly used for integrity checks and password storage, although weak hashing practices can still create security problems. The MD5 hash value 3b2a1053e3270077456a79192070aa78 resolves to 463729. This highlights that if weak or known values are hashed with outdated algorithms, attackers may still recover them through lookup tables or cracking.

Encoding, in contrast, is reversible. Its purpose is not to secure data but to convert it into a format suitable for transport or storage. Base64 is a common example because it converts data into plain-text characters that can be safely transmitted in systems expecting ASCII-compatible input. Because encoding is reversible, it should never be mistaken for encryption or protection. In the example shown, a session cookie value was Base64-encoded JSON. After decoding, the value revealed structured data showing a user ID and an admin flag. Since the contents were understandable and modifiable, the admin field could be changed and then encoded again to produce a new cookie value with elevated privileges. The Base64-decoded value of VEhNe0JBU0U2NF9FTkNPRElOR30= is THM{BASE64_ENCODING}, and the Base64 encoding of {"id":1,"admin":true} is eyJpZCI6MSwiYWRtaW4iOnRydWV9. The key idea is that encoded data remains readable to anyone who decodes it, so it offers no security by itself.

🔐 Overall Security Concepts

Authentication systems fail in more ways than simple password weakness. They can leak valid usernames through error messages, allow large-scale guessing through brute force, expose broken logic in reset workflows, and trust client-side cookies too much. Strong authentication security requires consistent error handling, protection against repeated guessing, strict validation of password reset flows, and server-side enforcement of identity and privilege. Any security decision based only on client input, whether in URLs, form fields, or cookies, is vulnerable to tampering unless the server validates it carefully. Hashing must be used appropriately, encoding must never be treated as protection, and authentication workflows must be designed so that attackers cannot redirect, override, or manipulate trusted account actions.
