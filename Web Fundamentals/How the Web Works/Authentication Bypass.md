🔐 Authentication Bypass

🔎 Authentication bypass is the process of gaining access to an application without completing authentication in the intended secure way. This usually happens because the application trusts user-controlled input too much, leaks useful information during login-related actions, or contains weak logic in its authentication workflow. These flaws are critical because they can lead to unauthorized account access, privilege escalation, and disclosure of sensitive information. The main areas involved here are username enumeration, brute forcing, logic flaws in password reset and access control processes, cookie tampering, and weaknesses involving hashing or encoding.

👤 Username enumeration happens when an application reveals whether a username is valid by responding differently to login, registration, or password reset requests. If one response says a username already exists and another says it does not, an attacker can confirm which accounts are real. This makes later attacks more efficient because the attacker only has to guess passwords for valid users instead of guessing both usernames and passwords. In this case, the valid usernames discovered were simon, steve, and robert. This kind of testing is commonly automated with ffuf, a web fuzzing tool that sends repeated requests while replacing a keyword such as FUZZ with entries from a wordlist. By comparing response size, wording, or status codes, ffuf makes it easier to identify the requests that behave differently and therefore expose valid usernames.

⚙️ A brute force attack is an automated attempt to guess passwords by trying many possible values until one works. Once valid usernames are known, brute forcing becomes much more effective because the attack is focused only on real accounts. If the application does not enforce rate limiting, account lockout, CAPTCHA, or multi-factor authentication, repeated attempts may eventually succeed. The valid credentials identified were steve/thunder. The same tool, ffuf, can also be used here by automating repeated login requests with password guesses and then filtering the responses to find the one that indicates success, such as a redirect, different content length, or a successful login message. This shows how small information leaks, such as confirming valid usernames, directly strengthen password attacks.

🧠 A logic flaw is a weakness in how an authentication process is designed rather than a simple coding error. In security, this means the workflow can be manipulated so the application behaves in an unintended but still technically valid way. One example is when access control only checks one exact path and fails to handle other equivalent variations. A more serious logic flaw appears in the password reset process, where one input is used to identify the account but a different input source controls where the reset link is sent. Because the application relies on PHP $_REQUEST, which combines GET and POST values, an attacker can supply a legitimate victim email in the URL for account validation and a different email in the POST body for reset delivery. This allowed the reset link for Robert’s account to be redirected and used to access his support tickets, resulting in THM{AUTH_BYPASS_COMPLETE}. This type of flaw is usually tested with Burp Suite, which acts as an intercepting proxy between the browser and the application. Burp allows requests to be captured, modified, and resent, which makes it possible to alter GET and POST parameters and observe how the server handles conflicting input.

🍪 Cookie tampering is the modification of cookie values stored in the browser in order to change how the application interprets the user’s session or privilege level. Since cookies are stored client-side, they should never be trusted for security decisions unless they are protected and verified server-side. If an application stores privilege information in plain text, an attacker may simply edit the values and resend them. In this case, cookies such as logged_in=true and admin=false were changed so the session was treated as an administrator session, revealing THM{COOKIE_TAMPERING}. This is commonly tested either with the browser’s Developer Tools, which allow inspection and editing of stored cookies in the Storage or Application tab, or with Burp Suite, which allows the Cookie header to be modified before the request reaches the server. The main lesson is that authorization decisions must never depend on client-editable state alone.

#️⃣ Hashing and encoding both transform data, but they serve different security purposes. A hash is a one-way transformation commonly used for password storage and integrity verification. It is not meant to be reversible, although weak algorithms and common inputs can still be cracked. The MD5 hash 3b2a1053e3270077456a79192070aa78 resolves to 463729, which shows why outdated algorithms such as MD5 are unsafe for sensitive data. This kind of lookup is often done with services such as CrackStation, which compare the hash against large databases of previously cracked or known values. These tools show that weak passwords and weak hash algorithms can often be recovered quickly if the value is common or predictable.

🔐 Encoding is different because it is reversible and is meant for formatting or transport, not protection. Base64 is a common example because it converts data into text-safe characters for transmission, but it does not secure the contents. In this case, a Base64-encoded value decoded to THM{BASE64_ENCODING}, and the JSON object {"id":1,"admin":true} encoded to eyJpZCI6MSwiYWRtaW4iOnRydWV9. This matters because if cookies or tokens are only Base64-encoded, a user can decode them, modify their contents, and encode them again. A tool such as CyberChef is commonly used for this because it allows quick decoding and re-encoding of Base64 and many other formats directly in the browser. This makes it especially useful for testing session values, cookies, and parameters that look encoded but are not actually protected.

🛡️ Authentication failures often come from weak design as much as weak credentials. An application may leak valid usernames through its messages, allow unlimited password guessing, mix trusted and untrusted inputs during password resets, or store authorization state in cookies the user can edit. Tools such as ffuf automate repeated requests for enumeration and brute force testing, Burp Suite exposes and manipulates live requests to reveal logic flaws, Developer Tools allow direct inspection of cookies and client-side state, CrackStation demonstrates the weakness of poor hashing, and CyberChef helps decode and re-encode structured values. The broader lesson is that authentication and authorization must always be enforced on the server with consistent responses, secure session handling, strong password storage, and carefully designed workflows that cannot be overridden by user-controlled data.
