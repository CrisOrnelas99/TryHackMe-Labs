📂 File Inclusion Vulnerabilities

🧩 File inclusion vulnerabilities happen when a web application uses user-controlled input to load or read files from the server without properly validating that input. This usually appears in URL parameters, form fields, cookies, or other request values that are passed into file-handling functions. If the application trusts the supplied path too much, an attacker may be able to read unintended files, include sensitive application resources, or even execute malicious code depending on how the vulnerable function behaves. The main categories here are path traversal, Local File Inclusion (LFI), and Remote File Inclusion (RFI).

🌐 These vulnerabilities exist because of poor input validation. The issue is not only the file-handling function itself, but the fact that user input is passed into it without strict filtering, sanitization, or allowlisting. When the application accepts file names or paths directly from the user, the attacker may submit values outside the intended directory or even point the application toward completely different resources. The overall risk includes sensitive information disclosure, exposure of source code, leakage of credentials or operating system files, and in some cases escalation into remote command execution (RCE) if the attacker can make the application process attacker-controlled code.

📍 Path traversal, also called directory traversal, is a vulnerability where an attacker moves outside the application’s intended directory by abusing path syntax such as ../. Each ../ means “move up one directory,” so repeated use can climb from the application folder to the filesystem root and then access operating system files. In Linux targets, a common test file is /etc/passwd, while in Windows targets common examples include boot.ini or windows/win.ini. The key idea is that the attacker is not limited to the application’s own files if the path is not properly constrained. The PHP function identified as causing the traversal issue here is file_get_contents.

📄 In simple path traversal cases, the application expects a file such as a PDF from a normal directory, but the attacker changes the parameter value to something like ../../../../etc/passwd to break out of that directory and read a system file instead. This works because the server interprets the path relative to the current location and follows it unless blocked by validation rules. The important concept is that traversal does not require code execution by itself; it already becomes dangerous if it allows reading files outside the intended scope, especially application configuration files, credentials, logs, or source code.

🛠️ Local File Inclusion (LFI) is closely related to path traversal but specifically involves functions such as include, require, include_once, and require_once in PHP. These functions do more than read a file; they bring the file into the current page execution context. If the input is not validated, the attacker may cause the application to include local operating system files or local application files that were never meant to be exposed. The vulnerable behavior is especially dangerous in PHP because including a file can affect how the page is processed, not just what content is displayed. LFI follows the same traversal principles but is often more powerful because of how inclusion functions behave.

🔓 In the first local inclusion lab, the request needed to read /etc/passwd is /lab1.php?file=/etc/passwd. This shows the most direct case, where the application accepts a file path with no directory restriction or meaningful validation. If the include or read function takes the parameter exactly as supplied, then any readable local file may be targeted. This is one of the clearest demonstrations of why user input must never directly control filesystem paths.

📁 Some developers try to reduce risk by forcing the application to prepend a directory before including the file. For example, instead of directly including the user’s value, the code may append it to a fixed folder like includes/ or another content directory. This is better than nothing, but it is still insecure if the attacker can use traversal syntax to escape that folder. In Lab #2, the directory specified inside the include function is includes, which means the attacker’s task is to break out of that directory and reach the target file instead.

⚠️ Error messages are extremely useful in black-box testing for LFI because they reveal how the server is building the file path internally. If invalid input causes the page to return a warning such as include(languages/THM.php), the attacker learns both the directory being used and any suffix being appended automatically, such as .php. Errors may also leak the full application path on disk, which helps estimate how many ../ sequences are needed to escape the current directory. This is why verbose PHP warnings and filesystem errors are considered dangerous disclosures in production systems.

🧪 One classic bypass shown here uses the null byte, represented as %00, to terminate the filename before an unwanted suffix such as .php is added. In older PHP behavior, if the application tried to build something like ../../../../etc/passwd%00.php, the null byte could cause the rest of the string to be ignored, effectively turning the include into a request for /etc/passwd. In Lab #3, the request used is /lab3.php?file=../../../../etc/passwd%00. A key point is that this technique is historically important but no longer works in newer PHP versions, so it is best understood as a legacy filter bypass rather than a modern universal trick.

🔍 Another filter bypass concept is that developers often block specific strings such as /etc/passwd or strip traversal sequences like ../, but do so incompletely. If the filter only removes the first matching pattern or performs a single pass, the attacker may still construct an input that resolves to the same final path after processing. For example, overlong traversal patterns such as ....// can survive weak replacement logic because the filter removes only one embedded ../ sequence and leaves the rest behind. The broader lesson is that blacklisting dangerous substrings is fragile because attackers can often rewrite the same path in alternate forms.

📚 Some filters can also be bypassed with path normalization tricks such as appending /. or /.. to a blocked filename. These rely on filesystem behavior, where . refers to the current directory and .. refers to the parent directory. If the application blocks a sensitive path literally but still passes the modified form to the filesystem, the operating system may normalize the path back into the intended target. In Lab #4, the function responsible for the traversal behavior remains file_get_contents, showing that reading functions are just as dangerous as include functions when they operate on untrusted paths.

🧱 In more restrictive cases, the application may require a certain directory name to appear inside the user input before it will process the request. This creates the illusion of safety, but if traversal is still possible, the attacker can include the required directory in the payload and then escape from it. In Lab #6, the required directory that had to be present in the input is THM-profile, and reading /etc/os-release from that lab reveals a VERSION_ID of 24.05. The deeper principle is that simple string-based requirements do not create real security if the filesystem path can still be manipulated afterward.

🌍 Remote File Inclusion (RFI) occurs when the vulnerable application allows a user-supplied value to point to a remote URL instead of a local file. If the server accepts a full external path in an include function, it may fetch and process content hosted by the attacker. This makes RFI more severe than ordinary LFI because the attacker is not just reading existing files; they may cause the server to execute malicious code obtained from a remote system. A major requirement in PHP is that options such as allow_url_fopen and related inclusion settings must permit remote file access for this to work.

💻 The danger of RFI is that it can lead directly to remote command execution, along with information disclosure, cross-site scripting (XSS), or denial of service (DoS) depending on how the injected file is used. A typical attack involves the attacker hosting a malicious PHP file on their own server, then supplying its URL to the vulnerable application. The application fetches that file and includes it as if it were trusted local content. In the practical RFI lab, the hostname command output obtained through this technique is lfi-vm-thm-f8c5b1a78692.

🧾 The challenge section reinforces that file inclusion entry points are not limited to GET parameters. They may appear in POST data, cookies, or HTTP headers, and the visible form field may not show the real request structure being used. This is why browser tools and interception tools are valuable: they let the tester modify request methods, parameters, cookies, and hidden values directly instead of relying only on the rendered page. The challenge flags recovered are F1x3d-iNpu7-f0rrn from /etc/flag1, c00k13_i5_yuMmy1 from /etc/flag2, and P0st_1s_w0rk1in9 from /etc/flag3, showing that inclusion flaws can be triggered across multiple request sources, not just URLs.

🛡️ The main remediation idea is to remove trust from user-controlled file paths entirely. Applications should use strict allowlists of approved filenames or resource identifiers rather than accepting arbitrary paths. Error messages should be disabled in production to avoid leaking internal directories, dangerous PHP settings such as remote URL inclusion should be disabled if not needed, and frameworks and services should be kept updated. A Web Application Firewall (WAF) can help reduce exposure, but the real fix is proper validation and safe application design. The important gap to fill here is that secure implementations should avoid direct path concatenation and instead map user choices to known-safe server-side resources.
