🔓 IDOR

🧩 IDOR stands for Insecure Direct Object Reference and is a type of access control vulnerability. It occurs when an application uses user-supplied input, such as an ID in a URL, form field, cookie, or request parameter, to access an object like a user record, file, document, or account data, but fails to verify on the server that the object actually belongs to the authenticated user making the request. The core problem is not that the identifier is visible, but that the application trusts it too much. Proper access control should always check whether the current user is allowed to access the requested object, regardless of what identifier is submitted. The answer to what IDOR stands for is Insecure Direct Object Reference.

🌐 A simple IDOR example is when a profile page uses a URL such as /profile?user_id=1305 to display account information. If changing that value to another number, such as 1000, shows another user’s data without authorization, the application is vulnerable. This happens because the server uses the supplied user_id to fetch data directly without confirming ownership. In secure design, the server should derive the correct user object from the authenticated session or at least validate that the requested record belongs to that user before returning it. The flag from the example website is THM{IDOR-VULN-FOUND}.

🔢 IDORs are often found in identifiers that appear directly in query strings, form data, cookies, AJAX requests, or API endpoints. Developers sometimes try to hide raw numeric IDs by encoding them before sending them to the client, but encoding is not a security control. It only changes the format of the data, not its trust level. A common example is Base64, which converts data into an ASCII-safe string. If an application passes an encoded identifier between requests, an attacker can often decode it, change the underlying value, re-encode it, and send it back to test whether access changes. The common type of encoding used by websites here is base64.

🛠️ When testing encoded IDs, a common workflow is to use a Base64 decoder and encoder to inspect and modify the value. Tools such as online Base64 utilities or CyberChef are useful because they let the tester decode the string, identify whether it contains a user ID or object reference, modify that value, and then encode it again for resubmission. The important concept is that if changing the decoded value changes which object is returned, the server is relying on client-supplied identifiers without proper authorization checks. That is what turns an exposed identifier into an actual IDOR issue.

#️⃣ Some applications use hashed IDs instead of plain or encoded values. A hash is more difficult to interpret directly because it is not meant to be reversible in the normal way, but it can still be insecure if it follows a predictable pattern. For example, if an application hashes simple integer IDs using a common algorithm, the values may still be guessable or searchable. The material highlights md5 as a common algorithm used for hashing IDs. If the underlying pattern is just “take the user number and hash it,” then the identifier is still predictable enough to support IDOR testing.

🔍 A useful tool for checking hashed identifiers is CrackStation or another hash lookup service. These tools compare a hash against large databases of known values and can reveal whether the hash maps to something simple, such as a low integer or common string. If a hashed ID turns out to be predictable, the tester can generate nearby values or look up additional hashes to see whether other users’ objects become accessible. This does not mean all hashes are insecure, but it shows that hashing alone does not solve access control problems if the application still trusts whatever identifier the client submits.

👥 If the identifier is not obvious as plain text, encoded, or hashed in a recognizable way, a strong method for detecting IDOR is to create two separate accounts and compare how object references behave between them. This helps determine whether one account can access the other account’s resources by swapping identifiers while remaining authenticated as the first user. The reason this works is that it isolates the access control question: if one valid user can retrieve another valid user’s content simply by changing an identifier, then authorization is broken. The minimum number of accounts needed to test for IDORs between accounts is 2.

📡 IDORs are not always visible in the browser’s address bar. Many vulnerable references are found in background requests, especially AJAX or API calls made after the page loads. Modern applications often fetch account details, profile data, order history, or messages through client-side scripts rather than rendering everything directly in HTML. That means important endpoints may only appear in the browser’s Developer Tools, particularly in the Network tab. Watching those requests can reveal parameters such as id, user_id, account, or other object references that are not shown directly in the page interface.

🔬 In the practical example, refreshing the account page while the browser’s Network tab is open reveals a call to /api/v1/customer?id={user_id}. This shows that the page is retrieving customer data from an API endpoint using a query string parameter. Because the returned JSON includes the user ID, username, and email address, the tester can infer that changing the id value may return another customer’s details if access control is weak. The browser’s developer tools are essential here because they expose the exact request path, the parameter being used, and the response format, which makes it possible to test for IDOR in a controlled way.

📨 The response format also matters when hunting IDORs. In this case, the endpoint returns JSON, which is common in APIs and client-side web applications. JSON responses are easy to inspect and often contain structured fields such as usernames, email addresses, or internal IDs. If changing a parameter causes the server to return another user’s JSON object without verifying ownership, that is direct evidence of broken object-level authorization. 🛡️ The main security lesson is that object references themselves are not the real issue; the real issue is missing server-side authorization. Numeric IDs, Base64 values, hashed identifiers, or hidden API parameters are all acceptable in an application as long as the server validates that the current user is authorized to access the requested resource. Encoding, hashing, or hiding identifiers in background requests may make them less obvious, but none of those methods replace access control. Secure design requires checking ownership or permissions every time an object is requested, no matter where the identifier came from.
