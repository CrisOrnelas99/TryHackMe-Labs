🌐 Walking a Web Application — Condensed Detailed Notes
🧭 Task 1 — Walking an Application

Walking an application is the process of manually exploring a web application to understand its structure, features, and possible weaknesses. This matters because automated tools often miss context, hidden functionality, and logic flaws that become clear through direct observation. The browser is not only used to view the website but also to analyze how the application behaves on the client side. The main built-in tools introduced are View Source, Inspector, Debugger, and Network. View Source shows the raw code returned by the server, Inspector shows the live rendered page after changes from CSS and JavaScript, Debugger allows JavaScript analysis and execution control, and Network displays the requests and responses exchanged with the server. Together, these tools help identify information disclosure, exposed functionality, client-side controls, and configuration mistakes. The main lesson is that manual review gives a fuller understanding of the application’s attack surface than surface-level browsing alone. The lab answer for this task is Done.

🔍 Task 2 — Exploring the Website

Exploring the website means mapping the application’s visible pages and interactive features to understand where input is accepted and where sensitive functionality may exist. Important areas include forms, buttons, login sections, links, and any part of the site that changes based on user interaction. These sections are valuable in security analysis because they often connect directly to backend actions, validation logic, or access control decisions. This stage is essentially reconnaissance: identifying what the application exposes, how pages connect, and what appears restricted or dynamic. JavaScript should also be reviewed during exploration because it may reveal hidden endpoints, client-side validation rules, or background actions not obvious from the interface. The main concept is that application mapping builds the foundation for later testing by showing where interaction happens and which components may become attack points. The lab answer for this task is Done.

📄 Task 3 — Viewing the Page Source

The page source is the code returned by the server to the browser and usually contains HTML, CSS, and JavaScript. It represents the original page content before runtime changes happen in the browser. Viewing page source is valuable because it can expose information not visible in the rendered page. One common source of disclosure is HTML comments, written between <!-- and -->. These are intended for developers and are hidden from normal page display, but they remain readable in the source. Comments may reveal internal notes, hidden routes, temporary pages, or development information. In the lab, a developer comment exposed a temporary homepage path, showing how comments can leak sensitive structure. This led to the first flag, THM{HTML_COMMENTS_ARE_DANGEROUS}.

Another useful part of source analysis is identifying links and hidden paths. Hyperlinks are defined with anchor tags, and the href attribute stores the destination. By reading the source, a tester may find routes that are not clearly presented in the visible interface. In the notes, a hidden link beginning with “secr” was visible in source code and led to another flag. This shows that not every reachable page is obvious through navigation, and source review can expose endpoints that users are not meant to notice easily. The flag found from this hidden link was THM{NOT_A_SECRET_ANYMORE}.

The page source also reveals external resources such as stylesheets, scripts, and images. These references expose the site’s file structure and can point to interesting directories. In the lab, external resources were stored in an assets directory, and directly visiting that directory revealed that directory listing was enabled. This is a misconfiguration because users should normally be blocked from browsing raw directory contents unless intended. When directory listing is active, files that were not meant for public viewing may become accessible, including backups, notes, archives, or internal resources. In this case, a flag.txt file was exposed. This issue reflects weak directory permissions or server indexing configuration. The flag obtained was THM{INVALID_DIRECTORY_PERMISSIONS}.

Page source can also help identify frameworks and versions. Many sites use frameworks that provide common web functionality, and comments or references in the source may reveal what framework is running and whether it is outdated. This matters because old versions may have publicly known vulnerabilities. In the notes, framework information and an update reference eventually led to a downloadable archive containing another flag. The important concept is that version disclosure can help determine whether software is behind on security updates. The flag gained from this finding was THM{KEEP_YOUR_SOFTWARE_UPDATED}. Overall, this task shows that source code review can expose hidden paths, comments, files, and technology details that support security testing.

🧪 Task 4 — Developer Tools: Inspector

The Inspector shows the live version of the webpage as it exists after HTML has been rendered and modified by CSS and JavaScript. This is different from page source, which only shows the original response from the server. The Inspector works with the DOM, the browser’s active representation of the page structure, so it reflects real-time content and styling. In security testing, this is useful because it reveals hidden elements, active overlays, and the exact structure currently shown in the browser. It also allows local editing of HTML and CSS, which helps test whether a restriction is truly enforced or only displayed in the interface.

In the lab, the Inspector was used on a premium article blocked by a paywall overlay. The blocking element existed as a visible layer in the page and was controlled with CSS. By editing the related style, specifically the display property, the overlay could be removed and the hidden content underneath became visible. This demonstrates a major security principle: frontend hiding is not real access control. If the content has already been sent to the browser, then CSS or JavaScript can only hide it visually, not protect it. Users can inspect and alter those controls themselves. Real protection must happen on the server by preventing unauthorized users from receiving the content at all. The flag revealed behind the paywall was THM{NOT_SO_HIDDEN}.

🐞 Task 5 — Developer Tools: Debugger

The Debugger, called Sources in some browsers, is used to inspect and control JavaScript execution. Since modern websites rely heavily on JavaScript for dynamic content and page behavior, this tool is important for understanding what the page is doing behind the scenes. It allows a tester to examine script files, reformat compressed code, and pause execution at specific lines. In the lab, a quick red flash appeared on the contact page, suggesting that JavaScript was briefly showing and then removing content. The Debugger made it possible to inspect the script responsible for that behavior.

A key concept in this task is minified and obfuscated JavaScript. Minification removes formatting to reduce file size, while obfuscation makes code harder to read. Neither is a real security measure because the code still runs in the user’s browser and can still be analyzed. The Pretty Print feature restores structure to minified code, making it easier to follow. The most important feature demonstrated is the breakpoint, which pauses execution at a chosen line. In the lab, a breakpoint was placed on the line that removed the red popup. After refreshing the page, execution stopped before the removal happened, so the hidden content remained visible. This revealed the flag THM{CATCH_ME_IF_YOU_CAN}. The main lesson is that JavaScript behavior can be observed and interrupted by the user, so sensitive logic should never rely on client-side secrecy.

🌐 Task 6 — Developer Tools: Network

The Network tab records the communication between the browser and the server and is one of the most useful tools for analyzing how a web application really works. Every page load or user action can trigger requests for documents, images, scripts, API responses, or form submissions. The Network tab shows these requests along with methods, URLs, headers, status codes, payloads, and responses. This helps reveal backend communication that may not be visible through the interface alone. In security testing, it is especially valuable for identifying hidden endpoints, understanding request flow, and inspecting the data being sent and returned.

In the lab, the contact form submitted data in the background without reloading the page. This behavior uses AJAX, which allows asynchronous communication between the page and the server. AJAX is commonly used for forms, dynamic updates, and interactive features because it lets the browser send and receive data without interrupting the current page. From a security perspective, AJAX requests are important because they often expose backend endpoints and response data that are not obvious in normal navigation. In the notes, submitting the form created a request named contact.msg, and the flag was visible in the response. The flag for this task was THM{GOT_AJAX_FLAG}. The key lesson is that network inspection often reveals more about application behavior than the visible page itself.

🔐 Overall Security Notes

This module shows that a large amount of security-relevant information can be found with only browser tools. Source code may expose comments, hidden links, file paths, and framework details. Directory listing can reveal sensitive files through server misconfiguration. The Inspector proves that content hidden with CSS or overlays is not actually protected if it already exists in the browser. The Debugger shows that JavaScript can be reformatted, paused, and studied even when compressed or obfuscated. The Network tab exposes the real communication between frontend and backend, including background requests and returned data. Across all tasks, the main security principle is that client-side content is fully exposed to the user. Real protection must be enforced server-side, sensitive information should not be embedded in frontend code, and exposed resources should be limited through proper configuration and updates.
