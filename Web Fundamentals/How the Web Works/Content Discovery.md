🔎 Content Discovery — Condensed Detailed Notes
📘 What Content Discovery Is

Content discovery is the process of finding files, directories, pages, features, and other resources on a web server that are not immediately visible through normal website navigation. In web application security, content does not only mean visible page text; it can include administration panels, staff portals, old versions of the site, backup files, configuration files, logs, media, and hidden application features. The purpose of content discovery is to reveal resources that may expand the attack surface or expose weaknesses. The notes group content discovery into three main methods: manual discovery, automated discovery, and OSINT. The answers for these are Manually, Automated, and OSINT.

🖐️ Manual Discovery

Manual discovery means checking likely locations and visible clues by hand rather than relying on automated enumeration tools. This method is useful because websites often expose information through standard files, metadata, and small implementation details. One important file is robots.txt, which is used to tell search engines which parts of a website should or should not be crawled. Although it is meant for search engine guidance, it can unintentionally reveal sensitive or less-public paths such as admin sections, customer areas, or internal portals. In the notes, the disallowed directory found in robots.txt was /staff-portal. This shows that files intended for indexing control can also leak useful reconnaissance data to an attacker.

Another manual method is checking the favicon, which is the small icon shown in the browser tab or address bar. Favicons can reveal framework or software information when developers leave the default icon from an installation unchanged. This becomes useful because identifying the framework helps narrow down the technologies in use and may lead to version research or known vulnerabilities. In the notes, the favicon hash matched the framework cgiirc, showing how a small branding file can become a clue about the application stack. This reinforces the principle that default assets and forgotten configuration details can reveal backend technologies.

A further manual technique is reviewing sitemap.xml. Unlike robots.txt, which suggests what should not be indexed, sitemap.xml lists content the website owner wants search engines to find. This file may expose pages that are difficult to reach through normal navigation, including older content or hidden sections still accessible on the server. In the notes, the secret path discovered in the sitemap was /s3cr3t-area. This shows that search engine support files can reveal paths that developers did not intend normal users to discover directly.

Manual discovery also includes reviewing HTTP headers. Whenever a client sends a request to a server, the response includes headers containing metadata about the response. These headers may reveal server software, framework information, scripting language versions, custom values, or internal debugging details. Such information can help identify outdated or misconfigured technology. In the notes, a custom response header exposed the value THM{HEADER_FLAG} through the X-FLAG header. This demonstrates that headers are not only protocol metadata but can also contain useful reconnaissance information or accidentally leaked internal details.

Another useful manual method is identifying the framework stack from page source, comments, credits, copyright text, or documentation references. Once a framework is known, its public documentation can reveal default paths, login panels, admin portals, and standard behavior. In the notes, the site source referenced a framework website, and the documentation disclosed the administration portal path. Accessing that portal on the target site revealed the flag THM{CHANGE_DEFAULT_CREDENTIALS}. The deeper lesson is that understanding the framework behind a target can lead to predictable content discovery because many frameworks include default routes and panels unless administrators change them.

🌍 OSINT Discovery

OSINT, or Open-Source Intelligence, refers to using publicly available external resources to gather information about a target. In web security, OSINT expands discovery beyond what the website directly reveals during browsing. One common technique is Google dorking, which uses advanced search operators to narrow results. A major operator is site:, which limits search results to a specific domain. In the notes, the answer to the operator that only shows results from a particular site is site:. This method is useful for finding admin pages, exposed files, cached documents, and indexed content that normal browsing may not show.

Another OSINT resource is Wappalyzer, which identifies website technologies such as frameworks, CMS platforms, analytics tools, payment systems, and sometimes version numbers. In the notes, the tool used to identify site technologies is wappalyzer. This kind of technology fingerprinting supports later research into known default paths, configuration patterns, and software weaknesses.

The Wayback Machine is another major OSINT source. It is a historical archive of websites that stores past snapshots of pages over time. This can help discover old content, removed sections, legacy pages, and paths that may no longer be linked but still exist on the current site. GitHub is also relevant to content discovery because developers may publish source code, project files, or configuration data in public repositories. ⚙️ Automated Discovery

Automated discovery is the process of using tools to send large numbers of requests to a web server in order to test whether specific files or directories exist. Instead of guessing paths one by one manually, automated tools use lists of common names and quickly check for valid responses. This is especially effective because websites often contain predictable file names, directories, backup files, logs, or default routes that are not linked from the main interface. Automated discovery is heavily dependent on wordlists, which are text files containing commonly used names for directories, files, parameters, or other targets. In the notes, wordlists are described as lists of commonly used words for different purposes, and the main resource mentioned is SecLists. For web content discovery, these lists contain frequent directory and file names used across websites.

The notes mention three common automation tools: ffuf, dirb, and gobuster. All three perform directory and file brute forcing by combining a base URL with entries from a wordlist and checking server responses. Their main purpose is to find resources that exist but are not linked publicly. Although the syntax differs, the concept is the same: automate repeated content checks at scale. In the practical results, the discovered directory beginning with “/mo” was /monthly, and the discovered log file was /development.log. These findings show why automation matters: hidden folders and log files may never be visible through ordinary browsing, but they can still be exposed if predictable names are used and proper access control is missing.

🔐 Overall Security Notes

This module shows that content discovery is a core reconnaissance activity in web security because hidden or unlinked resources often provide the most valuable findings. Manual discovery reveals useful information from standard files like robots.txt, sitemap.xml, page source, headers, and framework clues. OSINT extends discovery through search engines, technology fingerprinting, historical archives, public repositories, and cloud storage naming patterns. Automated discovery scales the process by testing large numbers of possible paths with wordlists and enumeration tools. Across all methods, the key principle is that content does not need to be visible in the interface to be reachable. Hidden paths, admin portals, backup files, logs, framework defaults, and cloud resources may still be accessible if they are not properly removed, restricted, or configured.
